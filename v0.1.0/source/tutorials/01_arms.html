Adding a robotic arm to the scene#

In the previous tutorial, we explained the basic working of the standalone script and how to play the simulator. This tutorial shows how to add a robotic arm into the stage and control the arm by providing random joint commands.

The tutorial will cover how to use the robot classes provided in Orbit. This includes spawning, initializing, resetting, and controlling the robot.

The Code#

The tutorial corresponds to the play_arms.py script in the orbit/source/standalone/demo directory.

 55def main():
 56    """Spawns a single arm manipulator and applies random joint commands."""
 57
 58    # Load kit helper
 59    sim = SimulationContext(physics_dt=0.01, rendering_dt=0.01, backend="torch")
 60    # Set main camera
 61    set_camera_view([2.5, 2.5, 2.5], [0.0, 0.0, 0.0])
 62
 63    # Spawn things into stage
 64    # Ground-plane
 65    kit_utils.create_ground_plane("/World/defaultGroundPlane", z_position=-1.05)
 66    # Lights-1
 67    prim_utils.create_prim(
 68        "/World/Light/GreySphere",

 71        attributes={"radius": 2.5, "intensity": 600.0, "color": (0.75, 0.75, 0.75)},
 72    )
 73    # Lights-2
 74    prim_utils.create_prim(
 75        "/World/Light/WhiteSphere",
 76        "SphereLight",
 77        translation=(-4.5, 3.5, 10.0),
 78        attributes={"radius": 2.5, "intensity": 600.0, "color": (1.0, 1.0, 1.0)},
 79    )
 80    # Table

 93    robot = SingleArmManipulator(cfg=robot_cfg)
 94    robot.spawn("/World/Robot_1", translation=(0.0, -1.0, 0.0))
 95    robot.spawn("/World/Robot_2", translation=(0.0, 1.0, 0.0))
 96
 97    # Play the simulator
 98    sim.reset()
 99    # Acquire handles
100    # Initialize handles
101    robot.initialize("/World/Robot.*")
102    # Reset states
103    robot.reset_buffers()
104
105    # Now we are ready!
106    print("[INFO]: Setup complete...")
107
108    # dummy actions
109    actions = torch.rand(robot.count, robot.num_actions, device=robot.device)
110    has_gripper = robot.cfg.meta_info.tool_num_dof > 0
111
112    # Define simulation stepping
113    sim_dt = sim.get_physics_dt()
114    # episode counter
115    sim_time = 0.0
116    ep_step_count = 0
117    # Simulate physics
118    while simulation_app.is_running():
119        # If simulation is stopped, then exit.
120        if sim.is_stopped():
121            break
122        # If simulation is paused, then skip.
123        if not sim.is_playing():
124            sim.step(render=not args_cli.headless)
125            continue
126        # reset
127        if ep_step_count % 1000 == 0:
128            sim_time = 0.0
129            ep_step_count = 0
130            # reset dof state
131            dof_pos, dof_vel = robot.get_default_dof_state()
132            robot.set_dof_state(dof_pos, dof_vel)
133            robot.reset_buffers()
134            # reset command
135            actions = torch.rand(robot.count, robot.num_actions, device=robot.device)
136            # reset gripper
137            if has_gripper:
138                actions[:, -1] = -1
139            print("[INFO]: Resetting robots state...")
140        # change the gripper action
141        if ep_step_count % 200 and has_gripper:
142            # flip command for the gripper
143            actions[:, -1] = -actions[:, -1]
144        # apply action to the robot
145        robot.apply_action(actions)
146        # perform step
147        sim.step()
148        # update sim-time
149        sim_time += sim_dt
150        ep_step_count += 1
151        # note: to deal with timeline events such as stopping, we need to check if the simulation is playing
152        if sim.is_playing():
153            # update buffers
154            robot.update_buffers(sim_dt)
155
156
157if __name__ == "__main__":
158    # Run the main function
159    main()
160    # Close the simulator
161    simulation_app.close()

The Code Explained#

Most of the code is the same as the previous tutorial. The only difference is that we are adding a robot to the scene. We explain the changes below.

Designing the simulation scene#

The single arm manipulators refer to robotic arms with a fixed base. These robots are at the ground height, i.e. z=0. Accordingly, the robots areplaced on a table that define their workspace. In this tutorial, we spawn two robots on two different tables. The first robot is placed on the left table /World/Table_1 and the second robot is placed on the right table /World/Table_2.

The tables are loaded from their respective USD file which is hosted on Omniverse Nucleus server

80    # Table
81    table_usd_path = f"{ISAAC_NUCLEUS_DIR}/Props/Mounts/SeattleLabTable/table_instanceable.usd"
82    prim_utils.create_prim("/World/Table_1", usd_path=table_usd_path, translation=(0.55, -1.0, 0.0))
83    prim_utils.create_prim("/World/Table_2", usd_path=table_usd_path, translation=(0.55, 1.0, 0.0))

Next, we create an instance of the single arm robot class. The robot class is initialized with a configuration object that contains information on the associated USD file, default initial state, actuator models for different joints, and other meta-information about the robot kinematics. The robot class provides method to spawn the robot in the scene at a given position and orientation, if provided.

In this tutorial, we disambiguate the robot configuration to load through the parsed command line argument. In Orbit, we include pre-configured instances of the configuration class to simplify usage. After creating the robot instance, we can spawn it at the origin defined by the tableâ€™s location.

Initializing the robot#

Prims in the scene with physics schemas enabled on them have an associated physics handle created by the physics engine. These handles are created only when the simulator starts playing. After that, the allocated tensor buffers used to store the associated primâ€™s state are accessible. This data is exposed through physics views that provide a convenient interface to access the state of the prims. The physics views can be used to group or encapsulate multiple prims and obtain their data in a batched manner.

Using the robot class in Orbit, a user can initialize the physics views to obtain views over the articulation and essential rigid bodies (such as the end-effector) as shown below. Multiple prims are grouped together by specifying their paths as regex patterns.

100    # Initialize handles
101    robot.initialize("/World/Robot.*")
102    # Reset states
103    robot.reset_buffers()

Running the simulation loop#

The robot class provides a method to obtain the default state of the robot. This state is the initial state of the robot when it is spawned in the scene. The default state is a tuple of two tensors, one for the joint positions and the other for the joint velocities. It is used to reset the robot to its initial state at a pre-defined interval of steps.

Note

Since the underlying physics engine in Isaac Sim does not separate the kinematics forwarding and dynamics stepping, the robotâ€™s state does not take into affect until after stepping the simulation.

126        # reset
127        if ep_step_count % 1000 == 0:
128            sim_time = 0.0
129            ep_step_count = 0
130            # reset dof state
131            dof_pos, dof_vel = robot.get_default_dof_state()
132            robot.set_dof_state(dof_pos, dof_vel)
133            robot.reset_buffers()

At the start of an episode, we randmly generate the joint commands for the arm. However, we toggle the gripper command at a regular interval to simulate a grasp and release action. The robot class provides a method to apply the joint commands. The type of command is configured in the robot configuration object.

140        # change the gripper action
141        if ep_step_count % 200 and has_gripper:

144        # apply action to the robot

After stepping the simulator, we can obtain the current state of the robot. The robot class provides a method to update the buffers by reading the data through the physics views. By default, the simulation engine provides all data in the world frame. Thus, the update method also takes care of transforming quantities to other frames such as the base frame of the robot.

The Code Execution#

Now that we have gone through the code, letâ€™s run the script and see the result:

./orbit.sh -p source/standalone/demo/play_arms.py --robot franka_panda

This should open a stage with a ground plane, lights, tables and robots. The simulation should be playing with the robot arms going to random joint configurations. The gripper, if present, should be opening or closing at regular intervals. To stop the simulation, you can either close the window, or press the STOP button in the UI, or press Ctrl+C in the terminal

In addition to the demo script for playing single arm manipulators, we also provide scripts for playing other robots such as quadrupeds or mobile manipulators. You can run these as follows:

 # Quadruped -- Spawns ANYmal C, ANYmal B, Unitree A1 on one stage
./orbit.sh -p source/standalone/demo/play_quadrupeds.py

# Mobile manipulator -- Spawns Franka Panda on Clearpath Ridgeback
./orbit.sh -p source/standalone/demo/play_ridgeback_franka.py

In this tutorial, we saw how to spawn a robot multiple times and initialize the physics views to access the simulation state of the robots. In the next tutorial, we will see how to simplify duplicating a simulation scene multiple times by using the cloner APIs.
